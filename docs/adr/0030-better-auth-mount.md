# ADR 0030 — Better Auth mounted in-trunk, same-origin, zero framework adapter

**Status:** Accepted (2026-05-30)
**Date:** 2026-05-29
**Deciders:** @numman (determination delegated to Claude Opus 4.8; review `wf_b3e0aac1-bff`)

## Context

Nomi's framing: *"Hono with Better Auth mounted, with a Next.js adapter at the mount — the API should ideally ONLY have the mount if Next.js is chosen."* ADR 0027 rejected Next.js, which **removes the conditional entirely**: there is no framework above the trunk, so there is no framework adapter to write. What remains is to fix where Better Auth lives and how its session reaches both the JSON-RPC surface and the collab WebSocket.

The original Next framing carried two now-moot hazards (project gotchas): `getServerSession` cannot be called inside a `"use cache"` scope, and `cookies()`/`headers()` cannot run inside `"use cache"`. With Next gone, both vanish.

## Decision

**Better Auth mounts directly on the Hono trunk** (its handler mounted at `/auth/*`), constructed once in the composition root (`getApiApp()`, ADR 0027/0029). No `@hono/*` framework adapter, no Next handler shim — the trunk *is* the server, so Better Auth's own request handler is the mount. The mount is a **side-effect on the trunk, deliberately kept out of the typed `AppType`** — `/auth/*` is not an `hc` capability route, so it must not contribute to the RPC client's inferred surface (ADR 0029 §8).

- **Same-origin → `SameSite=Lax`, no CORS.** The SPA, the JSON-RPC surface, and `/auth/*` are all on one origin (ADR 0027), so session cookies are first-party `Lax` cookies. No `SameSite=None`, no CSRF-token dance imported by a cross-origin split, no `trustedOrigins` allowlist beyond the canonical origin. This is the concrete same-origin dividend ADR 0027's topology buys.
- **One auth instance, shared with the dispatcher and the WS.** The Better Auth instance built in the composition root is the same one the capability dispatcher consults to resolve the principal (humans *and* agents — agents are first-class principals, ADR 0016) and the same one that validates the collab WebSocket's session cookie — read from the **HTTP upgrade request**, not from Hocuspocus's `token` (the verified mechanism is in Consequences). The session is resolved **once, in the trunk** — surfaces never re-implement permission/identity logic (invariant 5).
- **Published reader path needs no auth.** Custom domains (ADR 0011, via Caddy) serve the public static reader HTML, which carries no session — so multi-domain cookie scoping is a non-issue; auth cookies stay bound to the canonical app origin.

## Consequences

- **The "adapter at the mount" line item disappears, not simplifies.** The deliverable is "construct Better Auth in the composition root and mount its handler" — there is no adapter package, no conditional-on-framework branch. Nomi's "only the mount if Next" intent is satisfied by there being *only the mount*.
- **Cookie security posture is the simplest correct one.** `Lax` + first-party + `HttpOnly` + `Secure` (behind Caddy TLS). No exception surface to audit.
- **WS auth is real, the mechanism is specific, and the smoke proved it** (verified against `@hocuspocus/server` 3.4.4, the pinned version; `apps/server/src/cohost.integration.test.ts`). The collab session is **not** carried by Hocuspocus's `token`: `token` is opaque client data delivered *inside* a y-protocol Auth message *after* the socket is already open, and a browser cannot copy an `HttpOnly` cookie into it. The Better Auth session cookie rides the **HTTP upgrade request** instead, resolved through the one shared Better Auth instance — `getApiApp` now exposes `resolver` for exactly this. The control **splits across two points**, and the split is forced by Hocuspocus's model, not a preference:
  - **authN at the upgrade.** `server.on("upgrade")` reads the cookie from `request.headers.cookie`, resolves the principal via the shared `resolver`, and destroys the socket when there is none — rejecting unauthenticated clients *before* any WebSocket frame. The resolved principal is injected via `handleConnection(ws, request, { principal })` (possible only because we drive `handleConnection` ourselves; the built-in `Server` passes no context arg — source-confirmed).
  - **authZ at `onAuthenticate`.** Hocuspocus **multiplexes documents over one socket** — `documentName` is per-frame, auth is evaluated per document establishment — so per-document authorization must live where the doc name is known. `onAuthenticate` reads the injected principal from `payload.context` and authorizes `documentName` against the principal's workspace (a tenant-scoped doc lookup, invariant 5); a throw becomes Hocuspocus's permission-denied. Binding the first doc at the upgrade would not remove this — a multiplexed second `documentName` would still need the per-doc check, so one-doc-per-socket is at most optional defense-in-depth, not the gate.
  - **The gate is fail-closed — but only if `onAuthenticate` is registered.** Non-Auth frames are queued and never applied; a Document is established (and the queue flushed) *only* after a successful Auth frame, and a throw establishes nothing (`ClientConnection.ts:311-319`; sole establishment site `setUpNewConnection`). **The gate keys on the Auth frame, not on hook presence:** with no `onAuthenticate`, any Auth frame (even empty) establishes full read/write. "An `onAuthenticate` hook is registered and throws on deny" is therefore a **load-bearing invariant** with no in-package backstop. The production WS-attach pass (ADR 0027) enforces it **by construction** — a single `createAuthenticatedCollabHocuspocus` factory that always installs the hook, guarded by a negative test asserting a hook-less instance never reaches the production path — rather than by a runtime boot-time assertion (Codex red-team, 2026-05-31). The smoke always sets it.
  - **Auth state is per-connection** — a reconnect is a fresh socket and re-runs both checks, so the resolver and the authZ hook must tolerate being called on every (re)connect.

  This was the security-sensitive edge of the co-hosting story; the smoke exercised authenticated-accept, no-cookie-reject (at the upgrade), cross-workspace-deny (as Hocuspocus permission-denied), and **same-socket multiplex deny** (authenticate doc A, then an Auth frame for a cross-workspace doc B over the *same* socket is denied — pinning the per-document gate that forced authZ into `onAuthenticate`), so "embedded Hocuspocus on one port" is now demonstrated end-to-end on `@hono/node-server` **v1** — no v2 bump.
- **Production-attach hardening the smoke deliberately defers (Codex red-team, 2026-05-31).** The smoke's authZ is the tenant-isolation floor (the doc is visible only inside `principal.workspace_id`); three additions are required before write-capable collab ships, all on this ADR's auth boundary:
  - **Role-aware read/write, not just tenant membership.** `guest` carries `doc:read`/`block:read` but not `doc:write`/`block:write` (`packages/scopes`), yet Hocuspocus defaults every established connection to `readOnly: false` and accepts writes unless `connection.readOnly` is set. The attach must deny principals lacking the write scopes *or* set `payload.connectionConfig.readOnly = true` for read-only roles (with a write-rejected test). "Tenant membership ⇒ full collab write" is inconsistent with the role table.
  - **An `Origin` check on the raw WS upgrade.** Raw WebSockets sit outside Hono/Better Auth's CORS/trusted-origin path; `SameSite=Lax` mitigates cross-site WS hijacking but the upgrade gate should still reject browser upgrades whose `Origin` is not the canonical public origin (`originAllowed(req.headers.origin, public_origin)` + a wrong-origin/valid-cookie reject test). Absent `Origin` is permitted only if non-browser clients are intentionally supported on this cookie path.
  - **Revocation freshness — the injected principal is an upgrade-time snapshot.** `handleConnection(ws, req, { principal })` freezes membership/roles for the socket's lifetime, so a later per-document Auth frame still sees stale roles after a remove/demote. The attach must either re-validate membership/roles inside `onAuthenticate` per document or close affected sockets on `workspace.member_remove` / demotion / session revocation — folded into the ADR 0016 revocation bucket (invariant 8).
- **Agent principals authenticate differently from humans.** Agents use token/key credentials (ADR 0016), not the browser cookie flow; the dispatcher's principal resolution already unifies both. This ADR doesn't change agent auth — it ensures the *human* browser path is same-origin-simple and that both feed one resolver.

## Revisit triggers

- **The frontend must split to a different origin** (CDN-hosted SPA, separate auth domain): the `Lax`/no-CORS dividend is lost — re-introduce `trustedOrigins`, `SameSite=None`, and CSRF handling. This is downstream of ADR 0027's single-box revisit trigger.
- **A second auth method with a different cookie/redirect model** (SSO/OIDC, SAML) lands: re-confirm the same-origin assumptions and the WS `onAuthenticate` path hold for the new flow.

## Cross-references

- **Depends on** ADR 0027 (same-origin topology; the composition root + WS `onAuthenticate`), ADR 0011 (Caddy/custom domains — public, auth-free).
- **Feeds** the dispatcher's principal resolution (invariant 5, ADR 0016 agent principals) and ADR 0028's `beforeLoad` session guard.
- **Retires** the Next-only `getServerSession`-in-`"use cache"` and `cookies()`-in-`"use cache"` gotchas.

## Amendment — production WS attach landed; the deferred hardening is in (2026-06-12)

Task #15 closed the "production-attach hardening" deferrals above (design settled with Codex pre-build, 2026-06-12; both his MUST-FIXes are reflected below). The smoke's private bare-Hocuspocus wiring is **gone** — WS clients now attach to the *same* embedded Hocuspocus the dispatcher writes through, so live convergence (HTTP `ctx.transact` → broadcast to attached clients) is real and pinned. How each deferral resolved, plus the two postures that go beyond the original blocker text:

1. **Universal `readOnly`, not role-aware write (yet).** The blocker asked for "deny write-scope-less principals *or* readOnly for read-only roles". Slice A goes further: `HocuspocusSync`'s own registered `onAuthenticate` forces `connectionConfig.readOnly = true` for **every** principal *after* the policy passes — because invariant 3 (every mutation ⇒ exactly one audit row) is unsatisfiable over the WS write lane until WS-applied updates route through the audited dispatcher path. No policy outcome can grant WS write today; Hocuspocus enforces it server-side (Update/SyncStep2 from a readOnly connection are nacked, never applied — state-asserted in the tests, not just the denial frame). Role-aware write standing activates with the **audited WS write lane** (slice B, with broadcast-after-commit).
2. **Origin allow-list at the upgrade.** `attachCollab` (apps/server) refuses upgrades whose `Origin` doesn't match `config.public_origin`, compared **normalized** (`new URL(...).origin` both sides — config noise like a trailing slash can't break legit clients; folded/multiple/opaque/malformed values refuse). **Absent `Origin` refuses too**: every supported client on the cookie path is a browser, browsers always send it on WS upgrades, and absence-tolerance only shields non-browser clients that could fake the header anyway. Native/bearer clients get their own rule when that auth path lands.
3. **Revocation freshness = re-resolve per Auth frame.** Nothing identity-shaped rides the connection context anymore (the upgrade-time principal is only an early-refusal gate). `collabAuthorize` — composed in `getApiApp`, handed to `HocuspocusSync` as a constructor dep — re-runs the shared `BetterAuthResolver` against the upgrade request's headers **on every document Auth frame**, then authorizes with the gate's own terms reused, not re-implemented (invariant 5): `effectiveScopes(principal) ⊇ {doc:read}` + the Step-6 ceiling's `assertCanRead` on the live doc row. Session revoked mid-socket ⇒ the next document attach is denied (test: sign-out, then a new Auth frame on the same socket). *Already-established* doc subscriptions stay live until disconnect — acceptable **only because connections are readOnly**; event-driven closes on revocation remain in the ADR 0016 bucket and become mandatory with slice B's write lane.
4. **`onAuthenticate` by construction — stronger than the factory.** The recommendation above was a `createAuthenticatedCollabHocuspocus` factory. Landed shape: `HocuspocusSync` registers `onAuthenticate` **in its constructor, unconditionally**, with `collabAuthorize` an optional dep whose **default is deny-all**. A hook-less Hocuspocus (the "any Auth frame establishes full read/write" hole) is now unrepresentable on any path that uses `HocuspocusSync`, and a composition root that forgets the policy fails *closed* (every attach denied) instead of open. The dispatcher's `DirectConnection` lane never fires `onAuthenticate` (3.4.4: `openDirectConnection` builds its `connectionConfig` directly), so the hook cannot affect HTTP-path writes.
5. **Rollback eviction with WS clients (Codex MUST-FIX, found in his pre-build review).** Pre-existing recorded limit: with a WS client holding a doc resident, `bound.rollback()` could not evict, and the next `ctx.transact` would **persist deltas computed on aborted state** — a silent invariant-3/7 violation the moment any WS client attaches, not client cosmetics. `closeConnections` alone is *not* eviction (removal rides async close callbacks; `unloadDocument` refuses while connections or a pending debounced store exist). Landed: rollback force-closes the doc's WS subscriptions (per-doc Close frame; the multiplexed socket survives) and **awaits the unload drain**, actively driving `unloadDocument` as the count reaches zero; if eviction can't be proven inside the bounded window, the doc is **poisoned** and both open paths (`transact` + `read`) refuse loudly, retrying eviction under the doc lock until it succeeds. Deterministic tests pin the evict-and-rehydrate path (real WS client) and the poisoned/fail-closed/recover path (unevictable holder).
   - *Recorded residual (slice B):* the pre-commit broadcast already reached attached clients, and Yjs merges are additive — a client that applied the phantom delta keeps it locally until reconnect/resync. readOnly means it cannot be pushed back, and no production WS client exists yet (the SPA editor is HTTP-transport). The deep fix is **broadcast-after-commit**, which lands with the audited write lane.
6. **Cookie path admits user principals only.** The dispatcher's static `effectiveScopes` takes an *agent* token's scope claim verbatim — the live H8 `acting_as ∩ delegator` intersection lives in `workspaceAwareGate` — so reusing the static table for agents over WS would trust token claims. Cookie auth can't mint agent principals anyway; the bearer/agent WS path arrives with slice B and must factor the H8-aware term. Soft-deleted docs deny (restore first — ADR 0017's recovery capability is the sanctioned route back); malformed doc ids deny via the `DocId` brand parse.
7. **Serve-layer fold.** `attachCollab(server, booted, {publicOrigin, logger})` mounts on the raw `http.Server` through `startServer`'s new attachment seam; attachments close **first** in the drain (upgraded sockets count as open server connections — `server.close()` would otherwise wait on live WS clients forever; the cohost suite's teardown asserts this with a deliberately still-attached client). WS clients enter `HocuspocusSync` through the one public `handleWsConnection`, whose connection context carries only the `__ws` hydration marker — the `onLoadDocument` branch case 3 always reserved for this path, replaying committed `doc_updates` via the untransacted reader so cold attaches see durable state.

**Slice B (deferred, in order):** the audited WS write lane (WS-applied updates through the dispatcher's transact path — one audit row per mutation), broadcast-after-commit, role-aware write standing, event-driven revocation closes, bearer/agent WS auth with the H8-aware scope term.
