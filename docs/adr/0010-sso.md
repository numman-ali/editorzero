# ADR 0010 — Auth spine: Better Auth + SSO + OAuth Provider + MCP + API Key + Agent Auth

**Status:** Accepted (post-refresh, scope expanded from "SSO" to full auth spine)
**Date:** 2026-04-17 (v2)
**Deciders:** @numman

## Context
v1 scoped this ADR to SSO (`@better-auth/sso`). The April 2026 refresh revealed Better Auth now ships multiple first-class plugins (OAuth 2.1 Provider, MCP, API Key, Agent Auth) that consolidate work I had distributed across ADRs 0009, 0010, and 0016. Expanding this ADR to cover the full Better Auth adoption; the principal-model shape stays in ADR 0016.

Current Better Auth as of April 2026: **1.6.5** (April 16 2026). Pin **>= 1.6.5** to include:
- **GHSA-xr8f-h2gw-9xh6** fix (OAuth client-creation bypass in `@better-auth/oauth-provider`, 1.6.5).
- **OAuth state CSRF** fix (1.6.2).
- **Prototype pollution in Stripe plugin metadata merging** fix (1.6.3).

## Decision

**Adopt Better Auth as our entire auth spine.** Specific plugins:

| Plugin | Role | Notes |
|---|---|---|
| core `better-auth` | session / sign-in / sign-out / users | SQLite or Postgres adapter (dual-run = separate instances) |
| `@better-auth/sso` | OIDC + SAML 2.0, multi-tenant, per-workspace `providerId` | samlify >= 2.12; SAML SLO, signed AuthnRequests, `InResponseTo` + `AudienceRestriction` validated; replay protection always-on |
| `@better-auth/oauth-provider` | OAuth 2.1 Authorization Server — issues tokens for MCP clients and future external integrators | `validAudiences`, DCR (RFC 7591), PKCE S256 mandatory, offline_access gated |
| `@better-auth/mcp` | MCP resource-server middleware | `withMcpAuth` / `mcpAuthHono`; consumed by ADR 0009 |
| `@better-auth/api-key` | long-lived revocable scoped tokens | per-key `permissions`, `rateLimitMax`, `rateLimitTimeWindow`, `expiresIn`, `referenceId` (org/workspace) |
| `@better-auth/agent-auth` | **Agent Auth Protocol** (agent-auth-protocol.com) — distinct agent principal, `acting_as` delegation, device_authorization + CIBA | **treat as unstable** through 2026-H2; protocol in heavy iteration |

### Multi-tenant audience (DIY plumbing)
Better Auth exposes `referenceId` on clients and consents + `customAccessTokenClaims`, but **the "canonical URL per tenant" mapping is ours**. Implementation: a `resolveTenantAudience(host)` helper in the auth middleware resolves Host header → workspace → canonical MCP URL → audience claim. Called before issuing or validating tokens.

### Storage adapter
`@better-auth/kysely-adapter` for our chosen query layer (ADR 0007). SQLite + Postgres supported by picking adapter per instance; dual-run is separate Better Auth instances — one SQLite for single-node, one Postgres for scale. **Token portability between adapters is not first-class** and is our orchestration problem if we need it.

### Event / audit integration
Better Auth does not ship a structured event bus. Adapt via:
- `hooks.before` / `hooks.after` (post-commit) to emit our audit events.
- `@better-auth/agent-auth`'s `onEvent` hook for agent creation / revocation / host enrollment / capability requests / approvals / execution.
- `runInBackground()` / `runInBackgroundOrAwait()` for non-blocking side-effect pipelines.

Audit events land in our `audit_events` table (ADR 0016); Better Auth is a data source, not the audit system.

### Keys and rotation
`BETTER_AUTH_SECRETS` versioned keys support JWT-signing-key rotation without session invalidation. We rotate on a schedule (90d) and on suspected compromise.

### Step-up auth / incremental consent
Primitives exist (`WWW-Authenticate` on 401, scope-narrowed consent, `scopeExpirations`). **Full flow is DIY** — we compose it in our capability dispatcher when a capability requires a scope the current token lacks: return 403 with `WWW-Authenticate: Bearer error="insufficient_scope", scope="..."`.

## Consequences
- ~60% of what v1 ADR 0010 + ADR 0016 were going to hand-write is delegated to maintained upstream plugins.
- We pin, track advisories, and wrap — we don't re-implement OIDC/SAML/OAuth 2.1/DCR/PKCE/API keys.
- Per-tenant canonical URL handling is the one piece we own — it's worth owning because custom-domain multi-tenancy is product-specific.
- Agent Auth Protocol is still evolving; we adopt it but budget for breakage. Our capability dispatcher is insulated from Agent Auth specifics behind the `Principal` abstraction (ADR 0016).
- Dual SQLite + Postgres is not first-class in Better Auth; single-instance per deploy, mirroring our ADR 0007 posture.

## Revisit triggers
- Better Auth maintainership stalls (currently >11 releases in 8 weeks; healthy).
- Agent Auth Protocol 1.0 stable ships — pin to it.
- A CVE class we cannot pin through without a fork.
- Dual-backend token portability becomes a product requirement Better Auth does not support.
