- **2026-04-22** — **Slice-3 follow-on: `collection.restore` enforces the depth cap.** Closes a cross-slice correctness gap Codex caught reviewing slice 3: `collection.move`'s subtree-height walk only sees *live* descendants, so a "delete deep subtree bottom-up → move parent deeper → restore subtree top-down" sequence bypasses the cap — each individual restore looks cap-safe in isolation because the immediate subject has zero live subtree (per the `delete-refuses-with-live-descendants` invariant), but the deepest restore lands past `COLLECTION_MAX_DEPTH`. Exact minimal sequence: `A → B → C → D` live (depths 0..3) → delete D, C, B → move A under a parent at depth 4 (A now 5; move's subtree walk sees 0 live → passes) → restore B (at depth 6, OK) → C (7, OK) → D (8 → INVARIANT BROKEN). Fix: `collection.restore` now runs the same ancestor walk + BFS subtree walk `collection.move` uses and refuses with `ValidationError { depth_cap_exceeded }` when `parent_depth + 1 + subtree_height >= MAX_DEPTH`. Strict `>=` mirrors `collection.create` / `collection.move` exactly so no op that makes a collection live can produce a tree another would have rejected. Per the delete-live-descendants invariant the subtree_height should always be 0 on the normal path; the walk stays for defence-in-depth against future cascade-delete semantics or corrupt state. 3 new unit tests on `collection.restore` exercising the exact sequence (D-at-depth-8 refusal + B/C intermediate passes + root-restore-always-passes) + 1 new auth-chain e2e walking the full delete/move/restore trunk flow end-to-end and asserting 400 `validation_failed` on the final restore. 328 capabilities / 168 api-server / 99 CLI green. Invariant locality: every op that makes a collection live (create / move / restore) enforces `COLLECTION_MAX_DEPTH`.
